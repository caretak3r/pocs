Technical Implementation Strategy

1 The "Bundle" Concept

In a connected world, `helm install` reaches out to the internet for charts and Docker Hub for images. In an air-gap, this fails. The solution is to "vendor" everything.

# Example Zarf Definition (zarf.yaml)

kind: ZarfPackageConfig

metadata:

name: secure-app

components:

- name: app-charts

charts:

- url: https://charts.bitnami.com/bitnami

name: redis

version: 16.0.0

images:

- docker.io/bitnami/redis:6.2

Zarf will pull the chart, find the images in the chart, pull those images, generate SBOMs, and zip them all into one file.

2 Handling Metadata & Provenance

You need to ensure the file you transport across the air gap hasn't been tampered with. We use Cosign for this.

  • Step 1: Sign the images on the low side.
  • Step 2: Attach the SBOM as an attestation.
  • Step 3: Verify the signature inside the air gap before loading into the registry.
cosign sign --key cosign.key --attach-sbom=true my-registry/my-image:1.0

3 The Deployment (Helm + Registry)

Inside the air gap, you cannot reference `docker.io`. You must use a local registry (Harbor).

The Magic: Tools like Zarf run a "Mutating Webhook" inside your cluster. When you try to deploy a Helm chart that asks for `docker.io/nginx`, Zarf intercepts the request and rewrites it to `10.0.0.5:5000/nginx` (your air-gapped registry). This allows you to use standard Helm charts without modifying them manually.
